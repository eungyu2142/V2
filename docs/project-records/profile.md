# profile 작업 기록

## 2026-07-16 프로필 계정 표시 및 계정별 데이터 분리 기록

### 사용자 지적

프로필을 눌렀을 때 `"~~~"님` 형태로 현재 사용자를 보여주길 요청했다. 또한 다른 계정으로 로그인했는데 같은 계정처럼 똑같이 로그인/표시되는 오류가 발생했다고 지적했다.

### 원인

- 프로필 패널 상단이 고정 제목 `프로필`만 보여주고 있어 현재 로그인 계정 식별이 어려웠다.
- `loadAppData`가 `user_id` 조건 없이 테이블 데이터를 읽고 있었다.
- Supabase RLS가 완전히 분리하지 못하는 상황에서는 다른 계정의 펫, 기록, 초안 등이 섞여 보여 같은 계정처럼 보일 수 있었다.
- 임시저장 fallback localStorage도 공용 키를 사용해 계정 간 초안이 섞일 수 있었다.
- 계정이 바뀌는 순간 이전 계정의 화면 상태가 새 데이터 로드 전까지 남을 수 있었다.

### 수정 내용

- 프로필 패널 상단을 `{닉네임 또는 아이디}님`으로 표시하고 아래에 `프로필` 보조 라벨을 표시하도록 변경했다.
- `loadAppData`에 `userId`와 `scope: 'mine' | 'all'` 옵션을 추가했다.
- 마이펫, 다이어리 기록, 알림, 임시저장은 현재 로그인 사용자 `user_id`로 필터링해서 불러오도록 변경했다.
- QNA와 나눔은 전체 목록을 보되 row의 `user_id`를 기준으로 `mine` 값을 계산하도록 변경했다.
- `내가 쓴 글`은 `mine === true`인 QNA/나눔만 나오도록 변경했다.
- 임시저장 localStorage fallback 키를 `exocare:drafts:{userId}` 형태로 분리했다.
- 로그인 사용자 id가 바뀌면 이전 계정의 펫, 글, 임시저장, 프로필, 열린 패널/작성 상태를 먼저 초기화하고 새 계정 데이터를 다시 불러오도록 했다.
- 프로필 패널이 열린 상태에서 새 프로필 데이터가 도착하면 입력 필드도 새 프로필 값으로 동기화되도록 했다.

### 검증

- `npm run build`를 실행했고 TypeScript 및 Vite 빌드가 통과했다.

## 2026-07-13 프로필 사진, 내가 쓴 글, 좋아요한 목록 수정 기록

### 사용자 지적

프로필 사진이 URL 입력 방식으로 되어 있는데 사진 선택으로 바꿔야 한다고 지적했다. `내가 쓴 글`은 목록에 있기만 하지 말고 거기서 수정/삭제가 가능하고, 클릭 또는 열기 동작으로 해당 글 화면으로 이동해야 한다고 요청했다. `저장한 병원`이라는 표현도 맞지 않으니 전체를 `좋아요한`으로 바꾸라고 요청했다. 하단 탭도 너무 작아 조금 더 크게 보이길 요청했다.

### 수정 내용

- 프로필 사진 입력을 URL 입력에서 파일 선택 방식으로 변경했다.
- 선택한 사진은 Data URL로 미리보기와 프로필 저장에 사용한다.
- 프로필 메뉴의 `저장한 병원`을 `좋아요한`으로 변경했다.
- `좋아요한` 화면에 좋아요한 QNA와 나눔 글을 모아 보여주고, 열기 버튼으로 해당 글 화면으로 이동하도록 했다.
- `내가 쓴 글` 목록에 `열기`, `수정`, `삭제` 버튼을 추가했다.
- QNA 글은 프로필에서 열면 QNA 상세 화면으로 이동한다.
- 나눔 글은 프로필에서 열면 나눔 상세 화면으로 이동한다.
- 프로필에서 QNA/나눔 글을 수정하면 기존 글 id를 유지해 저장되도록 했다.
- QNA 수정 저장 시 기존 댓글, 좋아요, 해결 상태가 초기화되지 않도록 보존했다.
- 하단 탭의 높이, 글자 크기, 여백을 키웠다.

### 남은 연결

- 병원 자체 좋아요/저장 데이터는 기존 구현에 실제 저장 테이블/상태가 없고 placeholder만 있었기 때문에, 이번 작업에서는 프로필 메뉴명과 QNA/나눔 좋아요 목록을 우선 연결했다.
- 병원 좋아요까지 프로필 `좋아요한`에 넣으려면 지도 병원 카드에 좋아요 상태 저장과 DB 연결을 추가해야 한다.

### 검증

- `npm run build`를 실행했고 TypeScript 및 Vite 빌드가 통과했다.

## 2026-07-11 임시저장 범위 확장 기록

### 사용자 지적

임시저장이 QNA와 나눔에서만 되고 다른 작성 화면에서는 되지 않는다고 지적했다.

### 원인

- 공통 `DraftKind` 타입이 `question`, `share_item` 두 종류만 허용하고 있었다.
- 프로필의 이어쓰기 라우팅도 QNA 또는 나눔 작성 화면으로만 이동하도록 구현되어 있었다.
- 마이펫 등록, 다이어리 기록, 병원 리뷰 작성 화면에는 `drafts` 테이블 저장 버튼과 이어쓰기 초기값 복원 연결이 없었다.

### 수정 내용

- `DraftKind`에 `pet`, `care_record`, `hospital_review`를 추가했다.
- 프로필 임시저장 이어쓰기가 마이펫 등록, 다이어리 기록, 병원 리뷰 작성 화면으로도 이동하도록 분기했다.
- 마이펫 등록 화면에 `임시저장` 버튼을 추가하고, 이어쓰기 시 기존 입력값을 복원하도록 연결했다.
- 다이어리 기록 작성 화면에 `임시저장` 버튼을 추가하고, 이어쓰기 시 펫, 날짜, 기록 타입, 단계와 입력값을 복원하도록 연결했다.
- 병원 리뷰 작성 폼에 `임시저장` 버튼을 추가하고, 이어쓰기 시 해당 병원 패널과 리뷰 폼을 다시 열도록 연결했다.
- 알림 작성 화면에도 `임시저장` 버튼을 추가하고, 이어쓰기 시 알림 작성 화면을 다시 열도록 연결했다.
- 이어쓰기 완료 후 최종 저장하면 해당 draft를 삭제하도록 연결했다.

### 오류 및 검증

- 1차 빌드에서 `DiaryPage`의 `initialDraft` prop 타입이 `care_record`로 좁혀지지 않아 TypeScript 오류가 발생했다.
- App에서 `care_record` 분기 후 명시 캐스팅해 prop 타입을 맞췄다.
- 이후 알림 draft까지 추가한 뒤 `npm run build`를 다시 실행했고 TypeScript 및 Vite 빌드가 통과했다.

## 2026-07-11 프로필 임시저장 이어쓰기 수정 기록

### 사용자 지적

프로필의 임시저장 목록에서 초안 액션이 `등록`으로 표시되고, 초안을 바로 등록하는 흐름처럼 되어 있다고 지적했다. 임시저장은 게시가 아니라 작성 화면으로 돌아가서 `이어쓰기`를 해야 한다고 요청했다.

### 수정 내용

- 프로필 임시저장 목록의 액션명을 `등록`에서 `이어쓰기`로 변경했다.
- 기존 `publishDraft` 흐름을 제거하고 `continueDraft` 흐름으로 교체했다.
- 임시저장 초안을 클릭하면 프로필 패널을 닫고 QNA 또는 나눔 작성 플로우로 이동하도록 연결했다.
- QNA 작성 플로우가 draft payload의 제목, 본문, 사진, 펫, 카테고리, 연결 기록을 초기값으로 복원하도록 했다.
- 나눔 작성 플로우가 draft payload의 제목, 지역, 설명, 사진, 분류, 종, 성별 정보를 초기값으로 복원하도록 했다.
- 이어쓰기 중 다시 임시저장하면 새 초안을 만들지 않고 기존 draft id를 유지해 같은 초안을 갱신하도록 했다.
- 이어쓰기 완료 후 최종 저장하면 기존 draft를 삭제하도록 연결했다.

### 검증

- `npm run build`를 실행했고 TypeScript 및 Vite 빌드가 통과했다.

## 2026-07-10 프로필 임시저장 기능 기록

### 사용자 요청

프로필에 임시저장 기능을 추가하고, 프로필을 클릭했을 때 각 기능을 실제로 이용할 수 있게 연결해 달라고 요청했다. DB 연결이 되어 있다면 DB까지 연결되길 요청했다.

### 구현 내용

- `DraftItem` 타입을 추가했다.
- Supabase `drafts` 테이블을 `loadAppData`, `saveAppData`, `deleteAppData`로 연결했다.
- 로그인 사용자 기준으로 `drafts` 목록을 로드해 프로필 패널에 전달했다.
- QNA 작성 화면 마지막 단계에 `임시저장` 버튼을 추가했다.
- 나눔 작성 화면 마지막 단계에 `임시저장` 버튼을 추가했다.
- 임시저장 시 `draft_type`은 QNA는 `question`, 나눔은 `share_item`으로 저장한다.
- 프로필 패널에 `임시저장` 메뉴를 추가했다.
- 프로필 임시저장 목록에서 각 초안을 `등록`하거나 `삭제`할 수 있게 연결했다.
- `등록`은 저장된 payload를 실제 QNA 또는 나눔 DB 저장 함수로 넘긴 뒤 해당 draft를 삭제한다.
- `내가 쓴 글`, `저장한 병원`, `로그아웃`, `내 정보 수정` 메뉴도 클릭 시 패널 내부 화면이 바뀌도록 연결했다.

### DB 연결

- 사용 테이블: `public.drafts`
- 사용 컬럼: `id`, `user_id`, `draft_type`, `payload`, `created_at`, `updated_at`
- 기존 마이그레이션 `202606270002_feature_expansion.sql`에 `drafts` 테이블과 RLS 정책이 정의되어 있음을 확인했다.

## 2026-07-10 프로필 DB 연결 및 레이아웃 수정 기록

### 사용자 지적

프로필 패널에서 글자 간격과 배치가 이상하고, `내 정보 수정`이 실제 DB와 연결되어 있지 않은 것 같다고 지적했다.

### 원인

- `.profile-panel button`에 `width: 100%`가 적용되어 `뒤로` 버튼도 전체 폭을 차지했고, 그 결과 `프로필` 제목이 좁은 영역에서 세로로 줄바꿈되었다.
- `내 정보 수정` 화면은 실제 폼이 아니라 안내 문구만 표시하고 있었다.

### 구현 내용

- `AppProfile` 타입을 추가했다.
- Supabase `profiles` 테이블에서 현재 로그인 사용자의 `username`, `nickname`, `avatar_url`을 로드하도록 연결했다.
- `내 정보 수정` 화면을 실제 입력 폼으로 변경했다.
- 프로필 저장 시 `profiles` 테이블에 `id`, `username`, `nickname`, `avatar_url`을 upsert하도록 연결했다.
- 저장 성공 시 로컬 상태도 갱신하도록 했다.
- 프로필 사진 URL 미리보기 이미지를 추가했다.
- `뒤로` 버튼이 전체 폭을 차지하지 않도록 `.profile-back`의 폭과 flex 동작을 보정했다.
- 프로필 패널 폭을 넓히고 제목 줄바꿈을 막아 `프로필` 글자가 세로로 찢어지지 않게 했다.
- 프로필 입력 필드와 프로필 사진 미리보기 스타일을 추가했다.

### DB 연결

- 사용 테이블: `public.profiles`
- 사용 컬럼: `id`, `username`, `nickname`, `avatar_url`
- RLS 정책은 기존 마이그레이션에서 본인 프로필 select/insert/update가 허용되어 있음을 확인했다.
